# Product Requirements Document (PRD)
# Sistem Informasi Pemesanan dan Manajemen Villa (Versi Final)

## 1. Filosofi & Aturan Bisnis Inti (WAJIB DIIKUTI)
Sistem ini dibangun untuk mencegah kebocoran operasional akibat penerimaan uang tunai oleh pegawai. Oleh karena itu, AI yang membangun sistem ini wajib mematuhi aturan berikut:
1.  **100% Cashless (No Cash):** Tidak ada fitur pembayaran tunai. Semua transaksi wajib Transfer Bank ke rekening PT/CV.
2.  **Validasi Terpusat:** Hanya `Super Admin` yang boleh mengubah status pembayaran menjadi "Lunas" atau "DP Diterima".
3.  **Net Price:** Harga di database adalah harga bersih. Tidak perlu ada kalkulasi PPN/Service Charge di sistem.
4.  **Waktu Operasional:** Check-in pukul 14:00, Check-out pukul 12:00.
5.  **Tanpa Voucher Publik:** Tidak ada sistem input kode promo/voucher di halaman website publik.

## 2. Role-Based Access Control (RBAC)
Sistem menggunakan otorisasi ketat dengan 2 level pengguna:

### A. Super Admin (CEO / Manajer - Maksimal 2 Akun)
Pemegang otoritas tertinggi untuk urusan finansial.
*   **Validasi Uang (Eksklusif):** Satu-satunya *role* yang bisa mengubah `payment_status` transaksi.
*   **Pemberian Diskon (Eksklusif):** Hanya Super Admin yang bisa menginput nominal diskon (`discount_amount`) secara manual pada detail pesanan, khusus dilakukan saat proses pembuatan tagihan pelunasan (*Check-out*).
*   **Master Data:** Mengatur harga kamar, data kamar, menu makanan, dan rekening bank.
*   **Akses Laporan:** Mengakses seluruh analitik dan laporan keuangan.

### B. Admin (Pegawai / Resepsionis - Maksimal 2 Akun)
Fokus pada operasional fisik tamu tanpa akses validasi uang.
*   **Read-Only Keuangan:** Bisa melihat total tagihan dan status lunas/belum, tetapi TIDAK BISA mengubah harga atau memvalidasi pembayaran.
*   **Manajemen Tamu & Kamar:** Bisa mengubah status booking (Pending -> In-House -> Completed) dan status kamar (Cleaning -> Available).
*   **Input Pesanan Tambahan (Add-ons):** DIIZINKAN untuk memasukkan pesanan *Extra Bed* dan Makanan ke dalam tagihan tamu (*Running Tab*). Namun, validasi pembayaran atas pesanan ini tetap harus dilakukan oleh Super Admin.

## 3. Alur Kerja Sistem (System Workflow)

### Fase 1: Booking Web (Oleh Tamu)
1.  Tamu mencari kamar (Sistem mengecek ketersediaan dengan logika *overlap dates* pada tabel bookings. Kamar berstatus DP/Lunas pada tanggal tersebut disembunyikan).
2.  Tamu mengisi data diri, memilih metode mengetahui villa (Instagram/Google/dll untuk analitik).
3.  Sistem menghitung Grand Total (Harga Kamar) dan menampilkan kewajiban Minimal DP.
4.  Tamu menekan "Konfirmasi via WA" -> Diarahkan ke WA Villa mengirim bukti transfer.

### Fase 2: Validasi Booking Baru
1.  **Notifikasi Telegram:** Sistem (melalui *Cron* atau *Webhook*) mengirim alert ke Grup Telegram Super Admin: *"🚨 Booking Baru #VLA-001 dari [Nama]. Silakan cek mutasi DP."*
2.  Super Admin cek *mobile banking*, masuk ke sistem, klik "Terima DP". Status pesanan resmi aktif.

### Fase 3: In-Stay & Add-ons (Tamu Menginap)
1.  Tamu datang, Admin mengubah status tamu menjadi `In-House`.
2.  Tamu ingin *Extra bed* / Makanan. Admin membuka dashboard dan menginput pesanan tersebut ke data tamu.
3.  Karena aturan *Cashless*, tamu wajib transfer biaya makanan/extrabed tersebut. Tamu kirim bukti ke WA -> Super Admin memvalidasi pembayaran Add-ons -> Dapur menyiapkan pesanan.

### Fase 4: Checkout & Diskon (Pelunasan Akhir)
1.  Tamu ingin *check-out*. Jika ada sisa tagihan akhir, tamu memintanya.
2.  *Skenario Nego/Diskon:* Jika tamu bernegosiasi dan Ibu Tiwi (Super Admin) setuju, Super Admin membuka pesanan tersebut dan mengisi form "Nominal Diskon" (Misal: Rp 150.000). Sistem otomatis mengurangi sisa tagihan.
3.  Tamu transfer sisa pelunasan -> Super Admin klik "Lunas".
4.  Admin mencetak kwitansi (PDF), mengubah status tamu menjadi `Completed`, dan mengubah status kamar menjadi `Cleaning`.
5.  Setelah dibersihkan, Admin menekan "Kamar Siap", status kamar menjadi `Available`.

## 4. Analitik & Telegram Reporting
*   **Dashboard Super Admin:** Menampilkan (1) Pendapatan Kotor Bulan Ini, (2) Sisa Tagihan Belum Lunas, (3) Chart Okupansi, (4) Pie Chart Sumber Referensi Tamu.
*   **Telegram Auto-Report:** *Cron Job* mengirimkan rekap performa bulanan/mingguan ke Telegram Super Admin secara otomatis.

## 5. Arsitektur Database (Struktur Tabel Utama)
*Developer wajib menggunakan relasi Foreign Key yang rapi dan tipe data `DECIMAL` untuk kolom uang.*

*   **`users`**: id, name, email, password, role (`super_admin`, `admin`).
*   **`rooms`**: id, name, description, price, capacity, status (`Available`, `Cleaning`, `Maintenance`), image_path.
*   **`bank_accounts`**: id, bank_name, account_number, account_name, is_active.
*   **`bookings`**:
    *   id, booking_code, guest_name, guest_phone, acquisition_source.
    *   room_id, check_in_date, check_out_date.
    *   total_room_price, total_addons_price.
    *   `discount_amount` *(Hanya bisa diedit oleh Super Admin, default 0)*.
    *   grand_total, paid_amount, balance_due.
    *   payment_status (`Pending`, `DP`, `Lunas`, `Cancelled`).
    *   booking_status (`Booked`, `In-House`, `Completed`, `No-Show`).
*   **`booking_addons`**: 
    *   id, booking_id, item_name, type (`food`, `extrabed`), qty, price, subtotal.
    *   payment_status (`Pending`, `Paid`). *(Admin yang input data ini, Super Admin yang ubah statusnya jadi Paid).*
*   **`settings`**: id, key_name, value (Contoh: `min_dp_percent` = 50).

## 6. Spesifikasi Teknis Khusus (Tech Stack)
*   **Backend:** Laravel 13 (Arsitektur MVC, Fat Model - Skinny Controller).
*   **Frontend:** Blade Template, Tailwind CSS, Alpine.js (Responsive Mobile-First).
*   **Keamanan (Sangat Penting):** Buat *Middleware* khusus `role:super_admin` untuk memproteksi setiap *route* yang memproses pembaruan kolom uang (`paid_amount`, `discount_amount`, `payment_status`).
*   **Deployment:** Optimasi untuk cPanel. File system Laravel wajib di luar `public_html` kecuali isi folder `/public`.
